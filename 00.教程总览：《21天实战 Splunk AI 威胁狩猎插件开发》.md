# 🚀 教程总览：《21天实战 Splunk AI 威胁狩猎插件开发》

**课程目标**：指引零基础或有一定 Python 基础的安全运营人员，跨越 Splunk 底层架构的壁垒，开发出一款具备**依据 PEAK 威胁狩猎原理，融合 M-ATH 聚类发现、多维假设验证、自主下钻查询、API成本追踪、极简高管大屏**的企业级 AI 安全应用，并最终将其开源至 GitHub 及提交至 Splunkbase 官方插件商店。

***

## 🟢 第一阶段：环境初始化与大模型接入 (第 1-5 天)

*本阶段目标：搭建开发环境，打通 Splunk 与大模型的“任督二脉”，完成插件的骨架建设。*

* **Day 1：兵马未动，粮草先行 —— 搭建极客开发环境**
  * 安装 WSL (Windows Subsystem for Linux) 或 Linux 虚拟机。
  * 在 Linux 下通过命令行安装并启动 Splunk Enterprise 开发者环境。
  * 注册 Splunk 开发者账号并导入 License。
  * *达成目标：本地浏览器成功访问 Splunk Web 界面，获得企业级底层操作权限。*

* **Day 2：神器降临 —— 掌握 Splunk Add-on Builder (AOB)**
  * 下载并安装 AOB，深入理解 Splunk 插件的目录结构 (`default` vs `local`)。
  * 创建你的第一个插件项目，设定版本、图标与基本元数据信息。
  * *达成目标：在 AOB 中成功生成插件的底层架构骨架，为后续开发铺平道路。*

* **Day 3：打通 AI 经脉 —— 大模型 API 基础与测试**
  * 获取大模型 API Key（OpenAI、Anthropic、Ollama 或国内模型代理）。
  * 在外部独立编写 Python 脚本，测试请求大模型 API 的连通性。
  * *达成目标：熟悉大模型的入参 (messages) 与出参 (JSON) 结构，验证 AI 引擎可用性。*

* **Day 4：安全第一 —— 配置插件全局凭证 (Setup Parameters)**
  * 学习 AOB 的 UI 坑点：如何在 Data Input 界面正确切换到 Setup 页面。
  * 配置 `base_url`, `model_name` 和加密的 `api_key` 密码输入框。
  * *达成目标：掌握 Splunk 底层的加密密码保险箱机制，实现凭证的安全存储。*

* **Day 5：任务调度器 —— 建立后台数据输入流与心跳机制 (Data Inputs)**
  * 顺应 AOB 强制参数规则，改造默认字段为有意义的“心跳探针” (`custom_message`)。
  * 编写 Python 探针代码，配置时间窗口 (Interval) 与目标索引 (Target Index)。
  * *达成目标：配置出清爽的用户界面，并在 Splunk 底层索引中成功检索到按时触发的“心跳”日志。*

***

## 🔵 第二阶段：注入安全灵魂与底层引擎重塑 (第 6-10 天)

*本阶段目标：引入 PEAK 框架，跨越“聊天”界限，打造支持自动化执行、大屏监控与完美防崩溃结构的企业级智能体执行流。*

* **Day 6：连接数据之源 —— Splunk SDK 与 M-ATH 聚类实战**
  * 通过强壮方案跨越 AOB 上下文陷阱获取通信凭证 (`session_key`)。
  * 编写 M-ATH 算法 SPL，利用 `jobs.oneshot` 动态执行数据聚类，提取稀有日志模式（Rare Patterns）。
  * *达成目标：脚本彻底告别随机噪音，成功从海量数据中精准抓取具有分析价值的“孤岛”日志。*

* **Day 7：植入顶级大脑 —— PEAK 框架与多维 ABLE 假设工程**
  * 设定严酷的 System Prompt，强制大模型基于 ABLE 模型（Actor, Behavior, Location, Evidence）输出战术蓝图。
  * 引导 AI 生成包含 `Round 1 (验证)` 与 `Round 2 (下钻)` 的多轮 SPL 执行计划，并通过 Sandbox（沙盒）进行离线 JSON 格式验证。
  * *达成目标：大模型首次独立输出结构化狩猎策略，完成从“被动分析”到“主动狩猎”的跨越。*

* **Day 8：自主狩猎循环 —— 打造大屏友好的智能体执行引擎**
  * 编写 Python 循环结构，解析 AI 生成的 JSON 蓝图，并自动调用 Splunk SDK 静默执行多轮下钻 SPL。
  * 在循环中埋入“秒表”与“账本”，精准记录每次假设的执行耗时、证据命中数与 Token 分摊成本。
  * *达成目标：AI 智能体长出“手脚”，跑通自动化取证飞轮，为最终大屏奠定量化数据底座。*

* **Day 9：成果落地 —— Act 阶段战报生成与结构化落盘**
  * 触发二次大模型调用（Act 阶段），将 Day 8 收集的命中证据喂回给 AI，生成 0-100 风险打分与最终定性。
  * 组装庞大的包含计划、证据与评估的 Master Payload，并强制使用 `sourcetype="_json"` 写入系统索引。
  * *达成目标：Splunk 自动完成复杂 JSON 字段展平，成功输出可供高管审阅的企业级结构化战报。*

* **Day 10：直面深水区 —— 总结系统痛点与底层架构重塑**
  * **复盘痛点**：直面单一巨型日志（上帝对象）引发的“大模型 Token 风暴截断”与“Splunk 解析性能崩溃”灾难。
  * **代码重构**：引入 UUID 会话追踪，实施“扁平化革命”，将一个完整周期拆分为 `Plan`、`Evidence`、`Final_Report` 三个独立日志，并加入 `try...except` 残缺 JSON 防弹兜底逻辑。
  * *达成目标：完成从“能跑通的玩具脚本”到“坚不可摧、容错拉满的企业级引擎”的蜕变。*

***

## 🟣 第三阶段：灵魂注入与极致优化 (第 11-15 天)

*本阶段目标：彻底剥离 Mock 测试数据，接通真实云端大模型，并在真实极端的实战环境中完成极限抗压测试。*

* **Day 11：灵魂注入 —— 彻底废弃 Mock，接入真实大模型 API**
  * 移除 Day 8-10 中的硬编码测试数据，引入 `requests` 库完成真实网络通信封装。
  * 将 Day 6 抓取的真实异常日志动态注入 User Prompt，彻底激活智能体的实时分析能力。
  * *达成目标：Splunk 成功连通云端 OpenAI / 本地化大模型，根据当天真实日志现场编写并执行追踪语句！*

* **Day 12：动态上下文提纯 —— 极致优化 Prompt 与防护机制**
  * 针对真实数据往往过于庞大的问题，编写 Python 数据清洗与截断逻辑，防止超大日志撑爆 Prompt。
  * 在 API 请求中动态设定 `max_tokens`，并在 System Prompt 中加入极简表述指令压榨水分。
  * *达成目标：大幅降低 API 通信延迟，提升模型输出的稳定性与性价比。*

* **Day 13：精算师本色 —— 真实环境下的 API Token 成本提取**
  * 深度解析大模型真实返回 HTTP Header 或响应体中的 `usage.total_tokens` 字段。
  * 针对不同厂商 API（如 OpenAI, Anthropic, DeepSeek 等）格式不一的问题，编写强壮的正则/字典遍历提取函数。
  * *达成目标：将每一轮狩猎的真实资金成本精准记录进 Splunk，实现 FinOps 可审计化。*

* **Day 14：极限异常演练 —— 验证系统韧性与错误告警生成**
  * 人为制造灾难：断开服务器网络、输入错误的 API Key、使用极大文本触发 Token 截断。
  * 验证 Day 10 编写的兜底逻辑是否生效，观察系统是否能在崩溃边缘优雅退出并生成 `Error_State` 日志。
  * *达成目标：证明你的代码具备企业级鲁棒性，能够在极端情况下为 SOC 团队提供准确的事故现场。*

* **Day 15：底层大一统 —— 代码 Review 与 `transaction` 跨阶段数据校验**
  * 全面 Review Python 引擎代码，规范化日志输出层级 (`INFO`, `ERROR`, `DEBUG`)。
  * 在前端 Search 界面，使用高级 SPL 命令 `transaction session_id` 对最新产生的扁平化业务数据进行拼接校验。
  * *达成目标：后台核心 Python 引擎达到完美封版状态，数据基石坚若磐石。*

## 🟠 第四阶段：极客大屏开发与自动化运维 (第 16-18 天)

*本阶段目标：绕过前端工具的可视化局限，为管理层打造直观战报大屏，并建立底层的秒级恢复机制。*

* **Day 16：大屏呈现 —— 管理层视角 Dashboard 设计**
  * 抛弃繁杂的中间 SPL 细节，利用基于 `session_id` 的关联查询，重点提取分数、执行耗时与成本。
  * 构建“最近威胁狩猎报告汇总”高亮表格面板与“安全态势趋势图”，直观呈现风险打分与模型开销。
  * *达成目标：完成符合现代 SOC 负责人审美与汇报要求的炫酷 Dashboard XML 编写。*

* **Day 17：绕过 AOB 前端霸权 —— 底层文件系统注入术**
  * 深度剖析为什么 AOB 会无情覆盖自定义的 UI 配置文件。
  * 学习在 WSL 终端中使用 `vi` 编辑器的 `:set paste` 模式安全覆写 XML，规避致命的转义错误。
  * *达成目标：成功将极客大屏和自定义的顶层导航栏 (Nav) 强行且永久地注入到插件系统中。*

* **Day 18：从开发者到运维架构师 —— Shell 自动化脚本基建**
  * 编写 `backup_ui.sh` 脚本，将易被覆盖的配置文件安全备份出系统目录。
  * 编写 `restore_ui.sh` 脚本，实现一键静默覆盖，并利用 `curl` 带鉴权触发 `_reload` 端点刷新缓存。
  * *达成目标：实现 UI 环境配置的秒级“满血复活”，彻底消除开发后期的覆盖焦虑。*

***

## 🔴 第五阶段：开源、打包与官方发布 (第 19-21 天)

*本阶段目标：将个人心血转化为企业级公共资产，推向全球开源社区与 Splunk 官方商店。*

* **Day 19：安全第一的纯净打包法**
  * 破解 AOB 打包经常报错 `could not connect to App Certification` 的网络与底层环境原因。
  * 放弃 AOB 打包，使用 Linux 纯原生 `tar` 命令，严格 `--exclude` 掉包含隐私密钥的 `local` 目录和隐藏文件。
  * *达成目标：手动产出极其纯净、无毒无害、符合行业标准的 `TA-llm_analyzer-1.0.0.spl` 安装包。*

* **Day 20：拥抱开源 —— GitHub 顶级仓库建设与 Release 发布**
  * 在本地初始化 Git 仓库，编写严谨的 `.gitignore` 规则防止敏感信息泄露。
  * 撰写极具专业极客风的 `README.md` (包含 PEAK 架构图、严谨的安装指引、功能特性清单)。
  * *达成目标：将全部代码推送至 GitHub，发布闪亮的 v1.0 Release 版本供全球开发者克隆与下载。*

* **Day 21：进军殿堂 —— Splunkbase 上架与 AppInspect 严苛认证**
  * 学习使用 Splunk 官方的 AppInspect 工具（CLI版）在本地进行上架前的终极代码合规性扫描。
  * 在 Splunkbase 填写标准的插件元数据、上传高分辨率图标，并正式提交官方的安全与代码质量审核。
  * *达成目标：完成所有上架冲刺流程，静候插件在 Splunk 官方应用商店正式敲钟上线！*
