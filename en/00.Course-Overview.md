# 🚀 Course Overview: 21-Day Practical Guide to Developing a Splunk AI Threat Hunting Add-on

**Course Objective**: To guide security practitioners with zero or basic Python knowledge through the complexities of Splunk’s underlying architecture. You will develop an enterprise-grade AI security application that adheres to the **PEAK threat hunting methodology**, integrating **M-ATH clustering**, **multi-dimensional hypothesis validation**, **autonomous drill-down queries**, **API cost tracking**, and **minimalist executive dashboards**. Finally, the application will be open-sourced on GitHub and submitted to the official Splunkbase store.

***

## 🟢 Phase 1: Environment Initialization & LLM Integration (Day 1-5)

*Goal: Set up the development environment, establish communication between Splunk and the LLM, and build the skeleton of the add-on.*

* **Day 1: Logistics First — Building the Geek Development Environment**
    * Install WSL (Windows Subsystem for Linux) or a Linux VM.
    * Install and start the Splunk Enterprise developer environment via the command line.
    * Register a Splunk developer account and import the license.
    * *Goal: Successfully access the Splunk Web interface and gain enterprise-level underlying operational permissions.*

* **Day 2: The Divine Tool — Mastering Splunk Add-on Builder (AOB)**
    * Install AOB and understand Splunk add-on directory structures (`default` vs `local`).
    * Create your first add-on project, setting versions, icons, and metadata.
    * *Goal: Generate the architectural skeleton in AOB, paving the way for further development.*

* **Day 3: Opening the AI Meridians — LLM API Basics & Connectivity**
    * Obtain API keys (OpenAI, Anthropic, Ollama, or domestic proxies).
    * Write independent Python scripts to test LLM API connectivity.
    * *Goal: Familiarize yourself with request/response structures (messages/JSON) and verify AI engine availability.*

* **Day 4: Security First — Configuring Global Credentials (Setup Parameters)**
    * Learn AOB UI navigation: How to correctly switch to the Setup page.
    * Configure `base_url`, `model_name`, and encrypted `api_key` input fields.
    * *Goal: Master Splunk’s underlying password vault mechanism for secure credential storage.*

* **Day 5: The Task Scheduler — Establishing Background Data Inputs & Heartbeats**
    * Adapt to AOB's mandatory parameter rules and transform default fields into meaningful "Heartbeat Probes" (`custom_message`).
    * Write Python probe code, configure intervals, and target indexes.
    * *Goal: Configure a clean UI and successfully retrieve scheduled "heartbeat" logs in Splunk.*

***

## 🔵 Phase 2: Injecting the Security Soul & Underlying Restructuring (Day 6-10)

*Goal: Integrate the PEAK framework and move beyond simple "chatting" to build an enterprise agentic execution engine that supports automation, dashboard monitoring, and resilient data structures.*

* **Day 6: Connecting the Data Source — Splunk SDK & M-ATH Clustering**
    * Use a robust strategy to bypass AOB context traps and acquire communication credentials (`session_key`).
    * Write M-ATH (Model-Assisted Threat Hunting) SPL to execute data clustering via `jobs.oneshot`.
    * *Goal: Successfully extract 5 "island" logs (rare patterns) with high analytical value from massive noise.*

* **Day 7: Implanting the Top-Tier Brain — PEAK Framework & ABLE Hypotheses**
    * Set strict System Prompts to define a senior expert persona and enforce high-standard JSON output.
    * Guide the LLM to generate multi-dimensional ABLE (Actor, Behavior, Location, Evidence) hypotheses and multi-round drill-down SPL queries.
    * *Goal: The LLM independently outputs a structured hunting strategy, marking the transition from passive analysis to active hunting.*

* **Day 8: Autonomous Hunting Loop — Building a Dashboard-Ready Execution Engine**
    * Write Python loops to parse AI-generated JSON blueprints and automatically invoke the Splunk SDK for silent multi-round SPL execution.
    * Instrument code with "stopwatches" and "ledgers" to record execution duration, evidence hit counts, and token cost per hypothesis.
    * *Goal: The AI agent grows "hands and feet," completing the automated evidence collection flywheel.*

* **Day 9: Delivering Results — Act Phase Report Generation & Ingestion**
    * Trigger a second LLM call (Act Phase) to feed back evidence collected in Day 8 for final risk scoring (0-100) and qualification.
    * Assemble a "Master Payload" (Plan + Evidence + Assessment) and ingest it into the index using `sourcetype="_json"`.
    * *Goal: Splunk automatically flattens complex JSON fields, outputting a structured executive-ready report.*

* **Day 10: Navigating Deep Waters — Flattened Architecture & Resilience**
    * **The Retrospective**: Confront the "Token Storm" truncation and Splunk performance issues caused by massive single-line nested JSONs (God Objects).
    * **The Restructuring**: Introduce UUID session tracking and the "Flattening Revolution"—splitting the cycle into independent `Plan`, `Evidence`, and `Final_Report` events with robust `try...except` fallback logic.
    * *Goal: Complete the metamorphosis from a "toy script" to an indestructible, resilient enterprise-grade engine.*

***

## 🟣 Phase 3: Soul Injection & Extreme Optimization (Day 11-15)

*Goal: Completely strip away mock data, integrate real cloud LLM APIs, and perform extreme stress testing in real-world scenarios.*

* **Day 11: Soul Injection — Scrapping Mocks for Real LLM API Integration**
    * Remove hardcoded test data from Day 8-10 and implement the `requests` library for real network communication.
    * Dynamically inject real rare logs captured on Day 6 into the User Prompt to activate real-time analysis.
    * *Goal: Splunk successfully connects to cloud APIs (OpenAI/Qwen/etc.) to write and execute hunt queries on live data!*

* **Day 12: Dynamic Context Distillation — Optimizing Prompts & Guardrails**
    * Implement Python logic to clean and truncate real log data to prevent overwhelming the Prompt context.
    * Dynamically set `max_tokens` and add minimalist constraints to the System Prompt to reduce latency and cost.
    * *Goal: Significantly reduce API latency while improving response stability and cost-effectiveness.*

* **Day 13: FinOps Master — Real-World API Token Cost Extraction**
    * Deep-dive into parsing `usage.total_tokens` from real API response headers or bodies.
    * Build robust extraction functions compatible with various providers (OpenAI, Anthropic, DeepSeek, etc.).
    * *Goal: Permanently record the monetary cost of each hunting round into Splunk for auditable FinOps.*

* **Day 14: Extreme Failure Simulation — Verifying Resilience & Error Alerts**
    * Simulate disasters: Network disconnects, incorrect API keys, and Token limit overflows.
    * Verify the fallback logic from Day 10 to ensure the system exits gracefully and generates `Error_State` logs.
    * *Goal: Prove enterprise-grade robustness, providing SOC teams with an accurate "incident scene" during failures.*

* **Day 15: Underlying Unification — Code Review & Cross-Stage Validation**
    * Comprehensive code review of the Python engine, standardizing log levels (`INFO`, `ERROR`, `DEBUG`).
    * Use advanced SPL `transaction session_id` to validate the structural stitching of the latest flattened business data.
    * *Goal: Reach the final "Golden Baseline" for the backend Python engine.*

***

## 🟠 Phase 4: Geek Dashboard Development & Operations (Day 16-18)

*Goal: Bypass the visualization limitations of front-end tools to create intuitive dashboards for management and establish underlying recovery mechanisms.*

* **Day 16: Dimensional Reduction — Executive-Level Dashboard Design**
    * Use `session_id` correlation to extract risk scores, execution time, and costs, bypassing intermediate SPL details.
    * Build "Recent Hunt Report Summary" panels and "Security Posture Trends" to visualize risk and budget.
    * *Goal: Complete a stunning Dashboard XML that meets modern SOC leadership requirements.*

* **Day 17: Bypassing AOB Hegemony — File System Injection Techniques**
    * Analyze why AOB overwrites custom UI configurations and how to prevent it.
    * Use WSL terminal editors (`vi`) with `:set paste` to safely overwrite XMLs without escaping errors.
    * *Goal: Successfully and permanently inject geek dashboards and custom navigation (Nav) into the system.*

* **Day 18: From Developer to Ops Architect — Shell Automation Infrastructure**
    * Write `backup_ui.sh` to secure configuration files outside the system directory.
    * Write `restore_ui.sh` for silent restoration and use `curl` to trigger `_reload` endpoints to refresh the cache.
    * *Goal: Achieve second-level recovery of UI environments, eliminating "overwrite anxiety."*

***

## 🔴 Phase 5: Open Source, Packaging & Official Release (Day 19-21)

*Goal: Transform personal effort into enterprise-level public assets, reaching the global open-source community and Splunkbase.*

* **Day 19: Security-First Clean Packaging**
    * Troubleshoot why AOB often fails with `could not connect to App Certification` errors.
    * Ditch AOB packaging in favor of native Linux `tar` commands, strictly using `--exclude` for sensitive `local` directories and hidden files.
    * *Goal: Produce a clean, industry-standard `TA-llm_analyzer-1.0.0.spl` package.*

* **Day 20: Embracing Open Source — GitHub Repository & Release**
    * Initialize the local Git repo and write strict `.gitignore` rules to prevent credential leaks.
    * Write a professional `README.md` (including PEAK architecture diagrams and installation guides).
    * *Goal: Push code to GitHub and publish v1.0 for global developers to clone.*

* **Day 21: Entering the Hall of Fame — Splunkbase & AppInspect Certification**
    * Use the Splunk AppInspect CLI to perform final compliance scans.
    * Fill in metadata on Splunkbase, upload high-res icons, and submit for official security and quality review.
    * *Goal: Complete the submission process and wait for the official Splunkbase launch!*
